# Repository Guidelines

## Структура проекта и организация модулей
Основной код библиотеки находится в `src/`: публичная точка входа расположена в `src/index.ts`, вспомогательные проверки типов вынесены в `src/lib/`. Автотесты лежат в `test/`, входные фикстуры находятся в `test/geojson/in/`, ожидаемые результаты — в `test/geojson/out/`. Для ручной проверки используйте `playground/`: `playground/index.ts` для локального запуска и `playground/generate.py` для генерации данных. Каталог `dist/` содержит артефакты сборки и не редактируется вручную.

## Команды сборки, тестирования и разработки
- `npm test`: запускает тесты Mocha.
- `npm run lint`: проверяет код через ESLint с `--max-warnings=0`.
- `npm run lint:fix`: автоматически применяет безопасные исправления линтера.
- `npm run format`: форматирует `ts`, `js` и `json` файлы через Prettier.
- `npm run build`: последовательно запускает lint, тесты и сборку через `tsup`.
- `npm run watch`: пересобирает библиотеку при изменениях файлов.
- `npm run playground`: запускает локальный playground через `tsx watch`.

## Стиль кода и соглашения по именованию
Проект использует TypeScript в режиме ESM. Следуйте текущему стилю в `src/` и `test/`: отступ 2 пробела, одинарные кавычки, точки с запятой обычно опускаются, если инструмент не добавляет их сам. Публичный API держите в `src/index.ts`, а вспомогательную логику выносите в небольшие файлы внутри `src/lib/`. Для переменных и функций используйте `camelCase`, для типов GeoJSON — `PascalCase`. Имена фикстур должны явно описывать сценарий, например `smallPolygonNoCommonPositions.json`.

## Правила тестирования
Тесты написаны на Mocha с использованием `assert` из Node.js. Размещайте новые проверки по смысловым группам: валидация и опции — в `test/simplify.validation.test.ts`, поведенческие и инвариантные сценарии — в `test/simplify.behavior.test.ts`, упрощение без общих точек — в `test/simplify.no-common-positions.test.ts`, с общими точками — в `test/simplify.common-positions.test.ts`. Если меняется поведение библиотеки, синхронизируйте входные и ожидаемые JSON-файлы в `test/geojson/in/` и `test/geojson/out/`. Для новых нетривиальных GeoJSON-сценариев не инлайните большие объекты прямо в тесте: создавайте именованные фикстуры по существующему шаблону, например `polygonWithSmallInteriorRing.json` и `polygonWithSmallInteriorRingFraction0Tolerance1.json`. Предпочтителен один проверяемый сценарий на каждый `it(...)`, с названиями в стиле `should ...`. Перед открытием pull request обязательно запускайте `npm test`; для полной проверки используйте `npm run build`.

## Коммиты и pull request
История репозитория показывает короткие повелительные заголовки коммитов, например `Bump version`, `Update README.md`, `Fix package.json and bump version`. Придерживайтесь такого же краткого и предметного стиля. В pull request описывайте видимое изменение поведения, отмечайте изменения API или фикстур и прикладывайте ссылки на связанные issues, если они есть. Если меняется логика упрощения, добавляйте короткий пример GeoJSON до и после изменения.
